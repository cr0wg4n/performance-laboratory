# TypeScript Code Review Feedback
**Timestamp**: 2026-06-22 14:00:00
**Overall Quality**: ⚠️ Fair

## Issues Found

1. **`getUser` and `getMovements` do not check `response.ok`**
    - **Severity**: 🟠 Medium
    - **Description**: `createMovement` and `deleteMovement` correctly check `response.ok` and throw on failure. However `getUser`, `getMovements`, `getIncome`, and `getOutcome` call `.json()` unconditionally. If the server responds with a non-2xx status whose body happens to be valid JSON, the call returns corrupted data typed as `User | Movement[]` without any indication of failure. This silently corrupts reactive state in composables.
    - **Location**: `src/services/api.service.ts`, lines 6–22
    - **Recommendation**: Add `if (!response.ok) throw new Error(...)` after each `fetch` call, matching the pattern already used in `createMovement` and `deleteMovement`.

2. **`BASE_URL` is hardcoded instead of coming from an environment variable**
    - **Severity**: 🟠 Medium
    - **Description**: `const BASE_URL = 'http://localhost:3000'` is hard-wired, making it impossible to configure the API endpoint for staging or production builds without modifying source code. Environment-specific configuration should come from env variables, not from committed constants.
    - **Location**: `src/services/api.service.ts`, line 3
    - **Recommendation**: Use `const BASE_URL = import.meta.env.VITE_API_URL ?? 'http://localhost:3000'`. Declare `VITE_API_URL?: string` in the `ImportMetaEnv` interface in `env.d.ts` for full type-safety.

3. **Boolean variables `loading` not prefixed with `is`**
    - **Severity**: 🟠 Medium
    - **Description**: TypeScript and general JS conventions prefix boolean variables with `is`, `has`, or `can` to make their boolean nature immediately clear at the call site. `const { loading } = useMovements()` is ambiguous — it could be a loader object. `const { isLoading } = useMovements()` is self-documenting.
    - **Location**: `src/composables/useUser.ts` line 6; `src/composables/useMovements.ts` line 7; all consumers.
    - **Recommendation**: Rename `loading` → `isLoading` in both composables and propagate the rename through all consumers (`DashboardView`, `MovementsView`, `MovementList` prop, `MovementForm` prop).

4. **Three near-identical fetch functions in `useMovements` violate DRY**
    - **Severity**: 🟠 Medium
    - **Description**: `fetchMovements`, `fetchIncome`, and `fetchOutcome` share identical structure: set `isLoading`, clear `error`, await a service call, catch into `error`, and clear `isLoading`. Any future change to this pattern (e.g., adding a loading timeout or retry logic) must be replicated in all three places, creating a maintenance hazard.
    - **Location**: `src/composables/useMovements.ts`, lines 10–43
    - **Recommendation**: Extract a private `runFetch(fetcher: () => Promise<Movement[]>, errorMsg: string)` function that encapsulates the pattern. The three public functions delegate to it with their respective service call and error message.

5. **`MovementPayload` manually duplicates `Movement` fields**
    - **Severity**: 🟢 Low
    - **Description**: `MovementPayload` re-declares `type`, `amount`, and `description` — the exact fields that `Movement` has minus `id` and `date`. If `Movement` ever gains a new required field, `MovementPayload` must be updated manually. TypeScript's `Omit` utility eliminates this coupling.
    - **Location**: `src/types/index.ts`, lines 17–21
    - **Recommendation**: Replace the interface with `export type MovementPayload = Omit<Movement, 'id' | 'date'>`. This keeps the two types structurally linked automatically.

6. **Static route `props` object cannot be type-checked against the component's prop interface**
    - **Severity**: 🟢 Low
    - **Description**: Vue Router's `props: { type: 'income' }` is typed as `Record<string, unknown>` at the Router level — TypeScript cannot verify that the value matches `MovementsView`'s `Props` interface (`type: 'income' | 'outcome'`). A typo like `'icnome'` would not be caught at compile time.
    - **Location**: `src/router/index.ts`, lines 16–17 and 21–22
    - **Recommendation**: There is no perfect solution within Vue Router's current type system. As a pragmatic improvement, extract the prop values as `as const` typed constants: `props: { type: 'income' as const }`. This at least narrows the literal type within the router file and makes the intent explicit.
