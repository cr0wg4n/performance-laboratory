# Vue.js Code Review Feedback
**Timestamp**: 2026-06-22 14:00:00
**Overall Quality**: ❌ Poor

## Issues Found

1. **`deleting` ref in `MovementListItem` never resets after failed delete**
    - **Severity**: 🔴 High
    - **Description**: `handleDelete` sets `deleting.value = true` synchronously then fires `emit('delete', id)`. The parent handles the async API call. If the API call fails, the parent catches the error but the child `MovementListItem` component has no way to learn about the failure — `deleting` stays `true` permanently. The delete button remains disabled and shows a spinner indefinitely, locking the user out of retrying the action without a page reload.
    - **Location**: `src/components/movements/MovementListItem.vue`, line 20–23
    - **Recommendation**: Lift the loading/deleting state to the parent views via a `deletingId: number | null` ref. Pass it as a prop through `MovementList` to `MovementListItem` and compute `isDeleting = props.deletingId === props.movement.id`. The parent controls the full async lifecycle and can reset the ref in a `finally` block.

2. **`formError` persists when the modal is reopened**
    - **Severity**: 🟠 Medium
    - **Description**: After a failed submission, `formError` is set by `handleAddMovement`. When the user closes and reopens the modal, `showForm` transitions `false → true`, but `formError` is never cleared. The modal opens displaying the stale API error from the previous attempt, confusing the user before they have submitted anything.
    - **Location**: `src/views/DashboardView.vue`, line 66 (`@click="showForm = true"`)
    - **Recommendation**: Extract opening the modal into a `handleOpenForm` function that sets `showForm.value = true` and clears `formError.value = null`. Call this from the button's `@click`.

3. **Unhandled promise rejection in `MovementsView.handleDelete`**
    - **Severity**: 🟠 Medium
    - **Description**: `handleDelete` calls `await removeMovement(id)` and `await fetchUser()` without any `try/catch`. If either throws (e.g., server returns 404 or network error), an unhandled promise rejection is raised. The user gets no feedback and browser consoles log an error. Per the guidelines, side effects in composable-consuming code should handle errors gracefully.
    - **Location**: `src/views/MovementsView.vue`, lines 31–34
    - **Recommendation**: Wrap the body of `handleDelete` in a `try/catch` block and surface the error via a local `deleteError` ref shown as an alert in the template.

4. **`MovementForm` emits without client-side validation**
    - **Severity**: 🟠 Medium
    - **Description**: `handleSubmit` directly emits `submit` with whatever the form contains, including `amount: 0` (the default initial value) and an empty `description`. The server will reject these with 400 errors, but the round-trip is unnecessary. The user sees latency before the API error is shown. Client-side validation at the form boundary catches obvious errors instantly.
    - **Location**: `src/components/movements/MovementForm.vue`, lines 39–41
    - **Recommendation**: Add validation in `handleSubmit` before emitting: check `form.value.amount > 0` and `form.value.description.trim().length > 0`. Store validation messages in a local `validationError` ref and show it in place of (or alongside) `props.error`.

5. **Duplicate static and dynamic class cause incorrect style in `MovementsView`**
    - **Severity**: 🟢 Low
    - **Description**: The `card-body` div has the static class `text-success-content` hardcoded alongside a dynamic `:class` that conditionally applies `text-success-content` or `text-error-content`. The static class means `text-success-content` is always applied — even on the Expenses page where `text-error-content` should be the active color class.
    - **Location**: `src/views/MovementsView.vue`, line 40
    - **Recommendation**: Remove the static `text-success-content` from the element and rely solely on the dynamic `:class` binding.

6. **Empty `<style scoped>` tag in `App.vue`**
    - **Severity**: 🟢 Low
    - **Description**: An empty `<style scoped></style>` block adds no value and is noisy for readers scanning the file.
    - **Location**: `src/App.vue`, line 39
    - **Recommendation**: Remove the empty style block entirely.
